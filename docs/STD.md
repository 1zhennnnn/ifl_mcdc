**1　文件概述**

**1.1　目的與範圍**

本軟體測試文件（STD）依據 SRS v1.0 的 12 條功能需求（FR-01 ～ FR-12）及
SDD v1.0 的完整實作規格，制定涵蓋以下三個測試層次的完整測試方案：

- 單元測試（Unit Test,
  UT）：驗證每個類別與方法的獨立行為，包含正常路徑、邊界條件與異常路徑

- 整合測試（Integration Test, IT）：驗證三個功能層（Layer 1 → Layer 2 →
  Layer 3）之間的介面正確性與資料流

- 系統驗收測試（System Acceptance Test,
  SAT）：以完整疫苗篩選邏輯為基準，端對端驗證 IFL 迭代迴圈能在 50
  次內收斂至 100% MC/DC

> **🔑 重要　**本文件亦涵蓋 IEC 62304:2006 §5.5（軟體單元驗證）與
> §5.6（軟體整合測試）要求的測試活動對應關係，及 DO-178C Level A MC/DC
> 結構覆蓋率的量化驗收標準。

**1.2　術語與縮寫**

  ---------------- -----------------------------------------------------
  **縮寫**         **定義**

  **STD**          軟體測試文件（Software Test Document）

  **TC**           測試案例（Test Case）

  **UT**           單元測試（Unit Test）

  **IT**           整合測試（Integration Test）

  **SAT**          系統驗收測試（System Acceptance Test）

  **FR**           功能需求（Functional Requirement），編號 FR-01 ～
                   FR-12

  **MC/DC**        修正後條件/決策覆蓋率（Modified Condition/Decision
                   Coverage）

  **IFL**          迭代回饋迴圈（Iterative Feedback Loop）

  **SMT**          可滿足性模理論（Satisfiability Modulo Theories）

  **AST**          抽象語法樹（Abstract Syntax Tree）

  **L(X)**         損失函數：未覆蓋獨立對數量，L(X)=0 代表 100% MC/DC

  **Φ_gap**        SMT 缺口約束式，由 Z3 求解器針對未覆蓋獨立對合成

  **Ω**            可行解空間，滿足 Φ_gap
                   且符合醫療領域知識的輸入向量集合

  **PASS**         測試通過：實際結果與預期結果完全符合

  **FAIL**         測試失敗：實際結果與預期結果不符，需提交缺陷報告

  **NA**           不適用或待測

**2　測試策略**

**2.1　測試進入與退出條件**

**2.1.1　進入條件（Entry Criteria）**

- SDD v1.0 已發布且所有模組實作完成

- mypy \--strict 無 error（warning 可接受）

- 待測模組已通過 import 不崩潰的冒煙測試（smoke test）

- tests/fixtures/ 目錄完整：含 vaccine_eligibility.py、預期耦合矩陣
  JSON、預期探針日誌

**2.1.2　退出條件（Exit Criteria）**

- 所有 PASS/FAIL 測試案例均已執行，無「待測」

- FAIL 案例均已提交 Bug Report，並分派給對應開發人員

- UT + IT 程式碼覆蓋率 ≥ 80%，Layer 2 模組 ≥ 90%

- SAT 疫苗篩選端對端測試通過：L(X)=0 且迭代次數 ≤ 50

- IEC 62304 §5.5 / §5.6 查核清單所有項目打勾

**2.2　缺陷嚴重等級定義**

  ---------- ---------------- -------------------------------------------------------------
  **等級**   **名稱**         **定義與範例**

  **S1**     **系統崩潰**     測試執行導致系統崩潰、無限迴圈、資料損毀。例：ProbeInjector
                              改寫後模組無法 import

  **S2**     **功能失效**     核心功能完全無法使用。例：Z3 求解器對正確可滿足公式回傳 UNSAT

  **S3**     **部分失效**     功能異常但系統可繼續運行。例：MC/DC
                              矩陣漏標記一個已覆蓋的翻轉對

  **S4**     **輕微缺陷**     不影響功能的問題。例：日誌格式不符規格、錯誤訊息描述不清楚
  ---------- ---------------- -------------------------------------------------------------

**2.3　測試執行順序**

必須嚴格按照以下順序執行，因為後期測試依賴前期模組的正確性：

1.  Layer 1 單元測試（TC-U-01 ～ TC-U-18）

2.  Layer 2 單元測試（TC-U-19 ～ TC-U-36）

3.  Layer 3 單元測試（TC-U-37 ～ TC-U-56）

4.  Layer 1 + 2 整合測試（TC-I-01 ～ TC-I-04）

5.  Layer 2 + 3 整合測試（TC-I-05 ～ TC-I-07）

6.  全端系統驗收測試（TC-S-01 ～ TC-S-04）

> **⚠ 警告　**禁止在 Layer 1 單元測試全部通過之前開始 Layer 2
> 單元測試。錯誤的耦合圖會導致 SMT 合成出錯誤的約束，讓你完全搞不清楚是
> Layer 1 還是 Layer 2 的問題。

**附錄 A　pytest 執行指令參考**

以下為常用的 pytest 執行指令，請複製到終端機直接執行。

> \# 執行所有單元測試並輸出覆蓋率報告
>
> pytest tests/unit/ -v \--cov=ifl_mcdc \--cov-report=term-missing
> \--cov-fail-under=80

**A.2　分層執行**

> \# 僅執行 Layer 1 測試
>
> pytest tests/unit/test_layer1\_\*.py -v
>
> \# 僅執行 Layer 2 測試
>
> pytest tests/unit/test_layer2\_\*.py -v
>
> \# 僅執行 Layer 3 測試（使用 Mock LLM）
>
> pytest tests/unit/test_layer3\_\*.py -v

**A.3　整合測試**

> \# 執行所有整合測試
>
> pytest tests/integration/ -v

**A.4　系統驗收測試（需要 LLM API key）**

> \# 設定環境變數後執行 E2E 測試

>
