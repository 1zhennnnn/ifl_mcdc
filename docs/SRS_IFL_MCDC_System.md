# 1　文件概述

## 1.1　文件目的

本文件為「基於迭代回饋迴圈（IFL；Iterative Feedback Loop）之修正後條件 /
決策覆蓋率（MC/DC；Modified Condition/Decision
Coverage）優化機制系統」之軟體需求規格書（SRS）。本文件定義系統的功能需求、技術架構、驗收準則，以及與
IEC 62304:2006 與 RTCA DO-178C
國際法規標準的對應關係，作為後續系統設計、實作、測試與法規審查的基準文件。

## 1.2　文件範圍

本 SRS 涵蓋以下三個核心子系統的需求規格：

-   Layer 1｜靜態分析層：Python
    抽象語法樹（AST）解析、決策節點識別、布林條件分解、耦合圖建構、動態探針注入

-   Layer 2｜SMT
    推理層：布林導數計算、遮罩效應偵測、缺口損失函數評估、Z3
    求解器約束合成、可行解空間萃取

-   Layer 3｜LLM 協同層：缺口導向提示建構、醫療領域知識過濾、LLM
    採樣、接受門控與迭代回饋修正

本文件不涵蓋系統的 UI 介面設計細節，亦不包含部署基礎設施之規格。

# 2　系統背景與問題陳述

## 2.1　研究背景

隨著生成式 AI
與大型語言模型在軟體工程領域的應用日益普及，自動程式碼生成與自動測試生成已成為熱門趨勢。然而，目前的生成式測試技術多半停留在行覆蓋（Line
Coverage）或分支覆蓋（Branch
Coverage）的層次，對於安全導向之關鍵系統遠遠不足。

依循 IEC 62304:2006 國際標準的醫療器材軟體（Class C），以及依循 RTCA
DO-178C 的航太軟體（Level A），均要求達到 MC/DC
這一最高強度的結構化測試標準。MC/DC
要求每個邏輯條件都必須能獨立影響判定結果，這是純粹依賴 LLM
統計推論無法可靠達到的數學屏障。

## 2.2　核心技術瓶頸

### 2.2.1　遮罩效應（Masking Effect）

對於複合布林表達式，例如 (c₁ or c₂) and c₃，若測試案例生成時 c₃ =
False，則無論 c₁、c₂ 如何變化，整個判定結果恆為 False，導致 c₁、c₂
的獨立性完全無法被驗證。LLM
的統計推論本質傾向生成「語意合理的常見案例」而非能精準消除遮罩的邊界案例。

### 2.2.2　盲目迭代（Blind Iteration）

當 LLM
生成的測試集覆蓋率停滯時，傳統做法是重新提示（Re-prompt），但提示中缺乏明確的形式化診斷，使得
LLM
在已充分覆蓋的路徑上反覆生成重複案例，造成運算資源浪費，且無法保證收斂。

### 2.2.3　路徑爆炸（Path Explosion）

傳統搜尋基礎軟體測試（SBST）隨著條件數量 k 增加，搜尋空間呈指數級 2\^k
增長，在複雜醫療邏輯下迅速陷入計算瓶頸。

## 2.3　解決方案概要

本系統提出「混合式神經符號迭代優化架構」：以靜態程式分析精準定位遮罩成因（耦合圖），以
SMT 求解器將遮罩條件轉化為形式化約束
Φ_gap，再將此約束映射為自然語言的缺口導向提示，引導 LLM 在可行解空間 Ω =
{x \| Φ(x) ∧ Valid(x)} 中進行定向採樣。此架構不單純依賴 LLM
的黑箱生成能力，亦不依賴傳統 SBST 的盲目搜尋，而是三者的有機整合。

# 3　使用者需求分析

## 3.1　目標使用者

### 3.1.1　醫療軟體測試工程師（Primary Actor）

系統的主要操作者，具備軟體測試基礎知識，不一定深入理解 MC/DC
的數學推導。其核心痛點為：

-   手動撰寫 MC/DC 獨立對極度耗時，一個含 5 個條件的決策點理論上需要 10
    組精心設計的測試案例

-   因遮罩效應而漏測，且難以在不使用自動化工具的情況下自我發現

-   生成的測試資料常違反醫療常識（如年齡為負數），需耗費額外時間手動過濾

### 3.1.2　品質保證經理（Secondary Actor）

負責監控整個測試專案的合規進度，主要需求為：

-   即時查看 MC/DC 覆蓋率是否達到法規要求的 100%

-   評估迭代次數與測試效率，決定是否需要調配人力或延長測試時程

-   取得可直接提交 TFDA 或 CE 審查的結構化合規報告

# 4　系統架構概覽

## 4.1　三層神經符號架構

本系統採用「混合式神經符號迭代優化架構」，三個功能層形成一條線性處理管線，每一層的輸出是下一層的精確輸入：

  ----------- ------------ -------------------------- --------------------------------------------------------
  **Layer 1** 靜態分析層   解析 Python AST →          ast.NodeTransformer、ast.NodeVisitor、短路求值繞過機制
                           識別決策節點 →             
                           分解原子條件 → 建構耦合圖

                           MC/DC 覆蓋率矩陣 M         

  **Layer 2** SMT 推理層   計算布林導數 ∂f/∂xi →      Z3 SMT Solver、cvc5、Boolean Derivative、SMT-LIB 2.0
                           偵測遮罩效應 →             
                           評估損失函數 L(X) → Z3

  **Layer 3** LLM 協同層   Ω → Gap-Guided Prompt →    Llama 3 / GPT-4o、Chain-of-Thought、RLHF、HL7/FHIR 約束
                           LLM 定向採樣 →             
                           醫療領域過濾 → 接受門控 →  
                           迭代回饋修正 → 收斂判定    
  ------------------------------------------------------------------------------------------------------------

## 4.2　迭代回饋迴圈流程

整個系統的執行流程如下，共五個步驟，步驟 3 至 5 形成閉環：

1.  程式解析：上傳 .py 原始碼 → AST 解析 → 決策節點識別 → 耦合圖建構 →
    探針注入

2.  初始生成與評估：LLM 零樣本生成初始測試套件 → 計算基準 MC/DC
    覆蓋率矩陣 M

3.  缺口分析：計算 L(X) → 識別每個條件 ci 的未覆蓋獨立對 → Z3 合成 Φ_gap

4.  迭代優化：Φ_gap → Gap-Guided Prompt → LLM 生成 x_new → 領域驗證 →
    更新矩陣 M

5.  收斂判定：若 L(X)=0 → 停止並輸出報告；否則返回步驟 3

## 5.2　Layer 1：靜態分析層需求詳述

### FR-01　Python AST 解析與決策節點萃取

  -----------------------------------------------------------------------------------------------------------------------
  **屬性**       **內容**
  -------------- --------------------------------------------------------------------------------------------------------
  **需求描述**   系統應使用 Python ast 模組解析任意 .py 原始碼，走訪整棵 AST，定位所有 If、While、Assert 及行內三元表達式
                 IfExp 節點，並提取每個決策節點之完整布林表達式及其原始碼行號。

  **輸入**       原始 .py 原始碼檔案路徑（支援 Python 3.10+）

  **輸出**       DecisionNode 物件清單，每個物件含：{node_type, line_no, expression_str, source_context}

  **驗收準則**   （1）針對疫苗篩選模組，必須正確識別全部三個決策分支（年齡條件、間隔條件、過敏否定），零假陽性。（2）對
                 5,000 行以內的原始碼，解析時間不得超過 500 ms。（3）語法錯誤的原始碼應回傳結構化錯誤訊息而非崩潰。

  **合規對應**   IEC 62304:2006 §5.5.3 軟體單元驗證　DO-178C 結構覆蓋分析
  -----------------------------------------------------------------------------------------------------------------------

### FR-02　布林條件分解與耦合圖建構

  ------------------------------------------------------------------------------
  **屬性**       **內容**
  -------------- ---------------------------------------------------------------
  **需求描述**   對每個
                 DecisionNode，系統應遞迴分解複合布林表達式，提取原子條件集合 C
                 = {c₁, c₂, ..., cₖ}，並依 BoolOp
                 父節點關係建立耦合鄰接矩陣。耦合邊標記類型（OR-coupled /
                 AND-coupled），決定 SMT 求解器的約束方向。

  **輸入**       FR-01 產出的 DecisionNode 物件清單

  **輸出**       每個 DecisionNode 對應的 ConditionSet、k×k
                 耦合鄰接矩陣（含邊類型標記）、JSON 序列化輸出

  **驗收準則**   （1）對含 k 個條件的表達式，耦合矩陣中不得有虛假耦合邊（False
                 Positive）。（2）對 (c₁ or c₂) and c₃ and not c₄
                 表達式，必須正確識別 c₁↔c₂ 為 OR 強耦合，c₁↔c₃、c₃↔c₄ 等為 AND
                 弱耦合。（3）耦合矩陣必須序列化為合法 JSON 供下游 SMT 層使用。

  **合規對應**   為 FR-05 布林導數計算與 FR-07 Z3 約束合成提供輸入基礎
  ------------------------------------------------------------------------------

### FR-03　動態探針注入

  ---------------------------------------------------------------------------------------------------------------------
  **屬性**       **內容**
  -------------- ------------------------------------------------------------------------------------------------------
  **需求描述**   系統應以 ast.NodeTransformer 對原始碼進行 AST 重寫，在每個原子條件 cᵢ 的求值位置插入 probe(cond_id,
                 expr) 呼叫。探針必須在複合表達式組合執行之前完成所有原子條件的求值，以完全繞過 Python
                 短路求值（Short-circuit Evaluation）的截斷效應。

  **輸入**       原始 .py 檔案、ConditionSet（含各 cᵢ 的 AST 節點位置）

  **輸出**       儀表板化（Instrumented）.py 檔案；執行期探針日誌 Schema：{test_id, cond_id, value: bool, decision:
                 bool, timestamp}

  **驗收準則**   （1）儀表板化模組必須通過原始單元測試，確保控制流與語意未被改變。（2）即使在短路求值發生的情況下（如
                 c₁=True 導致 or 截斷），c₂
                 的探針仍必須觸發並記錄其真值。（3）探針執行開銷不得使原始程式碼執行時間增加超過 15%。

  **實作說明**   probe(cond_id, value) 函式設計為：記錄 (cond_id, value) 至執行緒安全佇列後，直接 return
                 value，確保對程式語意的零干擾。詳細 AST 重寫範例：\_c1 = probe(\'D1.c1\', Age \>= 65)；\_c2 =
                 probe(\'D1.c2\', HighRisk)；if (\_c1 or \_c2) and \_c3 and not \_c4: ...
  ---------------------------------------------------------------------------------------------------------------------

### FR-04　MC/DC 覆蓋率矩陣計算

  --------------------------------------------------------------------------------------------------------------
  **屬性**       **內容**
  -------------- -----------------------------------------------------------------------------------------------
  **需求描述**   執行測試套件後，系統應建構覆蓋率矩陣 M ∈ {0,1}\^{2k×m}，其中 2k 列對應所有條件的翻轉對（F→T 與
                 T→F 各一列），m 行對應測試案例。M\[i\]\[j\]=1 當且僅當測試案例 j 的探針日誌，滿足第 i
                 列翻轉對所需的獨立性條件（條件翻轉、決策翻轉、其他條件無影響）。

  **輸入**       FR-03 探針日誌、已執行的測試套件 T

  **輸出**       覆蓋率矩陣 M；覆蓋率純量（已覆蓋列數 / 2k）；未覆蓋獨立對清單 Δ

  **驗收準則**   （1）每新增一個測試案例，矩陣更新時間複雜度為 O(k·m)。（2）系統必須正確區分唯一原因型（Unique
                 Cause）與遮罩型（Masking）獨立對。（3）矩陣支援差異比較（diff），用於視覺化呈現迭代收斂進度。
  --------------------------------------------------------------------------------------------------------------

## 5.3　Layer 2：SMT 推理層需求詳述

### FR-05　布林導數計算與遮罩效應偵測

  -----------------------------------------------------------------------------------------
  **屬性**       **內容**
  -------------- --------------------------------------------------------------------------
  **需求描述**   SMT 推理引擎應計算布林導數 ∂f/∂xi = f(...,xi=T,...) ⊕ f(...,xi=F,...)。當
                 ∂f/∂xi = 0 時，條件 xi
                 在當前輸入下被遮罩。系統應識別造成遮罩的具體原因（即哪些其他條件的值導致
                 xi 無法影響決策），並輸出結構化的 MaskingReport。

  **輸入**       布林表達式 AST、當前測試輸入向量 x

  **輸出**       MaskingReport：{condition_id, is_masked: bool, masking_cause:
                 \[condition_id\], derivative_value: 0\|1}

  **驗收準則**   （1）對 (A∨B)∧C∧¬E，當 C=False 時，系統必須正確識別 A 和 B 被 C 遮罩，並在
                 masking_cause 中記錄
                 C。（2）布林導數計算必須精確（無近似），不得使用採樣或蒙地卡羅估計。

  **合規對應**   IEC 62304:2006 §5.5.3；為 FR-06 損失函數計算提供遮罩分類輸入
  -----------------------------------------------------------------------------------------

### FR-06　獨立對缺口分析與損失函數評估

  -----------------------------------------------------------------------------------------------------------------------------
  **屬性**       **內容**
  -------------- --------------------------------------------------------------------------------------------------------------
  **需求描述**   系統應評估損失函數 L(X) = Σᵢ∈C I(Pair(cᵢ) ∉ X)，精確計算當前測試集 X
                 中缺少多少個獨立對。對損失非零的每個條件，估計其滿足難度（基於需要同時約束的其他條件數量），輸出按難度排序的
                 GapList。

  **輸入**       FR-04 覆蓋率矩陣 M、FR-02 ConditionSet（含耦合鄰接矩陣）

  **輸出**       損失純量 L(X)；GapList：\[{condition_id, flip_direction, missing_pair_type, estimated_difficulty}\]（按
                 estimated_difficulty 升序排列）

  **驗收準則**   （1）L(X) = 0 若且唯若所有 2k 列均被覆蓋。（2）估計難度排名需與觀察到的 SMT 求解時間呈正相關（Pearson r ≥
                 0.6）。（3）系統優先攻克難度低的缺口，以最小化總迭代次數。
  -----------------------------------------------------------------------------------------------------------------------------

### FR-07　Z3 SMT 約束合成（核心機制）

  ---------------------------------------------------------------------------------
  **屬性**       **內容**
  -------------- ------------------------------------------------------------------
  **需求描述**   對 GapList 中的每個缺口，系統應將獨立對需求編碼為 SMT 公式
                 Φ_gap，呼叫 Z3 求解器判定可滿足性（SAT）或不可滿足（UNSAT）。SAT
                 時，Z3 回傳一個具體的輸入向量（可行解）；UNSAT
                 時，將該路徑標記為不可行（Infeasible），永久停止對此路徑的迭代。

  **輸入**       GapList 條目、布林表達式 SMT 編碼、領域型別定義（如 Age: Int,
                 HighRisk: Bool）

  **輸出**       SMT 結果：SAT with model（具體數值 Ω）或 UNSAT with infeasibility
                 certificate

  **驗收準則**   （1）對條件數 k ≤ 8 的表達式，Z3 求解需在 10 秒內完成。（2）UNSAT
                 結果需以否定假設進行第二次求解驗證。（3）已標記為 Infeasible
                 的路徑不得再次進入求解流程，防止無窮迴圈。

  **合規對應**   RTCA DO-178C DO-333 形式化方法補充文件；SMT
                 輸出可作為形式化分析之補充證據
  ---------------------------------------------------------------------------------

### FR-08　可行解空間萃取與邊界正規化

  --------------------------------------------------------------------------------------
  **屬性**       **內容**
  -------------- -----------------------------------------------------------------------
  **需求描述**   從 Z3 的滿足模型萃取每個涉及變數的可行輸入空間
                 Ω，將數值型約束正規化為閉區間 \[min,
                 max\]，類別型約束表示為枚舉有效集合。同時保留醫療單位標注（如
                 days、years）。

  **輸入**       FR-07 Z3 模型輸出

  **輸出**       BoundSpec JSON：{var_name, type, interval_or_set, medical_unit}

  **驗收準則**   （1）萃取邊界必須是 Φ_gap
                 數學解空間的嚴格子集或相等。（2）任何變數不得出現空區間，除非該路徑為
                 UNSAT。（3）BoundSpec 格式需通過 JSON Schema 驗證，供 FR-09 直接消費。
  --------------------------------------------------------------------------------------

## 5.4　Layer 3：LLM 協同層需求詳述

### FR-09　約束至自然語言缺口導向提示建構

  -----------------------------------------------------------------------------------------------------------------------------------------------------------
  **屬性**       **內容**
  -------------- --------------------------------------------------------------------------------------------------------------------------------------------
  **需求描述**   提示建構模組應將 FR-08 的 BoundSpec
                 轉化為結構化自然語言指令，包含四個必要段落：（1）醫療情境描述；（2）目標條件獨立性說明；（3）各變數的精確數值約束；（4）輸出格式規範（JSON
                 物件符合函式簽名）。

  **輸入**       BoundSpec（FR-08 輸出）、模組說明文字（docstring）、領域情境字串

  **輸出**       結構化提示字串 P_prompt，遵循 Gap-Guided Prompting 範本

  **驗收準則**   （1）提示必須包含 BoundSpec 的所有數值約束，不得模糊化（如不得將 Age ∈ \[18,64\] 改寫為「年齡適當」）。（2）提示長度不得超過 2,048
                 tokens。（3）提示中的輸出格式規範必須對應目標函式的 Python 參數簽名。
  -----------------------------------------------------------------------------------------------------------------------------------------------------------

### FR-10　LLM 引導測試案例採樣

  ------------------------------------------------------------------------------
  **屬性**       **內容**
  -------------- ---------------------------------------------------------------
  **需求描述**   LLM 採樣模組應以 P_prompt 呼叫配置的語言模型 API（支援 Llama
                 3、GPT-4o、Claude 3.5 等）並解析結構化 JSON 回應。系統將 LLM
                 視為條件機率採樣器：x_new ∼ P_M(x \| Context,
                 Ω)。若解析失敗，以帶有原始回應與 JSON Schema
                 的元提示（Meta-prompt）重試，最多三次。

  **輸入**       P_prompt（FR-09 輸出）、LLM API 設定、重試策略

  **輸出**       符合目標函式輸入 Schema 的測試案例物件 x_new

  **驗收準則**   （1）不需重試的一次成功解析率 ≥ 85%。（2）每次 API 呼叫的 Token
                 消耗量需記錄，用於效率追蹤。（3）所有 API 錯誤（超時、rate
                 limit）需以指數退避（Exponential Backoff）重試。
  ------------------------------------------------------------------------------

### FR-11　醫療領域知識約束過濾

  ----------------------------------------------------------------------------------------------------------
  **屬性**       **內容**
  -------------- -------------------------------------------------------------------------------------------
  **需求描述**   系統應對 LLM
                 生成的每個測試案例執行可配置的領域規則集驗證，拒絕醫療上不合理的數值。預設規則集包含：age ∈
                 \[0, 130\]、LastDoseDate ≤ today、互斥欄位（如 pregnant=True 且
                 sex=Male）、字串欄位符合允許醫療詞彙。

  **輸入**       LLM 生成的 JSON 測試物件（FR-10 輸出）、DomainRuleSet 設定檔

  **輸出**       ValidationResult：{passed: bool, violations: \[{field, rule, actual_value}\]}

  **驗收準則**   （1）必須拒絕 age=-5、age=200、LastDoseDate 在未來的資料。（2）不得拒絕合法邊界案例如 age=0
                 或 age=130。（3）違規報告須作為修正上下文回饋給下一輪 LLM 提示（FR-10 重試機制），而非單純丟棄。

  **合規對應**   HIPAA 合規意識；IEC 62304 資料合理性要求；防止語意幻覺（Hallucination）污染測試集
  ----------------------------------------------------------------------------------------------------------

### FR-12　接受門控與迭代回饋修正

  ------------------------------------------------------------------------------
  **屬性**       **內容**
  -------------- ---------------------------------------------------------------
  **需求描述**   執行 x_new 後重新評估 L(X ∪ {x_new})。若 L
                 下降則接受並更新測試集；若 L
                 不變（未觸及目標缺口），將失敗原因（SMT
                 約束違反、非預期遮罩等）編碼為修正提示，返回 FR-09
                 重新生成。迭代持續至 L(X)=0 或預算耗盡。

  **輸入**       x_new（FR-11 輸出，已通過領域驗證）、更新後的探針日誌、迭代計數器

  **輸出**       更新後的測試集 X；收斂狀態：{CONVERGED \| BUDGET_EXHAUSTED \|
                 INFEASIBLE}；逐迭代損失日誌

  **驗收準則**   （1）L(X ∪ {x_new}) \< L(X) 時，必須在一次評估週期內接受
                 x_new。（2）已知 Infeasible
                 路徑不得再次進入迭代。（3）對基準疫苗篩選模組（k=5），達到 100%
                 MC/DC 的迭代次數不得超過 50
                 次。（4）預算耗盡時，必須輸出含剩餘缺口列表的部分覆蓋報告。

  **合規對應**   IEC 62304:2006 §5.6
                 軟體整合測試；確保自動化合規測試的完整性與可重現性
  ------------------------------------------------------------------------------

## 附錄 A　疫苗篩選基準邏輯

本系統以下列醫療邏輯作為核心驗證基準：

> def check_vaccine_eligibility(age, high_risk, days_since_last,
> egg_allergy):
>
> if ((age \>= 65 or (age \>= 18 and high_risk))
>
> and (days_since_last \> 180)
>
> and not egg_allergy):
>
> return True \# 符合接種資格
>
> return False

此表達式含 5 個原子條件（k=5），需要 2k=10 個翻轉對方可達到 100%
MC/DC，是系統複雜度的基準測試標的。
